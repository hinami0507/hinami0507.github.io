以后怎么发展

get和post

GET一般用于获取/查询资源信息,更小更快，但不安全，
而POST一般用于更新资源信息，更大更安全，

GET 的内容可以被浏览器缓存，而 POST 的数据不可以。

GET

GET一般用于获取信息而不是修改信息，就像数据库查询一样，不会修改和增加数据，不会影响资源状态，是幂等的（多次请求得到的结果都是相同的，因为没有改变服务器端的资源）以?分割URL和传输数据，参数之间以&相连
但是GET不是很安全，如果用get提交用户的登录的账号密码，那么账号密码就会出现在URL上，
(1)GET 的内容可能被浏览器缓存，
(2)其他人查看浏览器的历史纪录，那么别人就可以拿到你的账号和密码了，

在FORM（表单）中，Method默认为”GET”

POST

post只是把数据封装在表单中，也是可以被看到的，用加密技术就好了

http状态码

1XX：信息状态码

- **100 Continue**：客户端应当继续发送请求。这个临时相应是用来通知客户端它的部分请求已经被服务器接收，且仍未被拒绝。客户端应当继续发送请求的剩余部分，或者如果请求已经完成，忽略这个响应。服务器必须在请求万仇向客户端发送一个最终响应
- **101 Switching Protocols**：服务器已经理解为客户端的请求，并将通过Upgrade消息头通知客户端采用不同的协议来完成这个请求。在发送完这个响应最后的空行后，服务器将会切换到Upgrade消息头中定义的那些协议。
  • 2XX：成功状态码
  • 3XX：重定向
  • 4XX：客户端错误
  • 5XX: 服务器错误

用过什么第三方库，如何评价

jQuery最少的代码做最多的事情

优点：

  • 1、行为层与结构层的分离，例如标签中的onclick
  • 2、强大的选择器和DOM封装操作，$(),—使得操作DOM的时候不用新建变量来获取DOM节点
  • 3、性能优化好，兼容性好；
  • 4、不污染顶级变量。jQuery只建立一个名为jQuery的对象，其所有的方法都在这个对象之下
  • 5、本来是非官方标准，但业内对其支持已经很广泛，google也在用

  • 完善的Ajax.之前写ajax要考虑很多兼容性问题，但是jquery将所有ajax的操作封装到$.ajax，方便使用不需考虑XMLHttpRequest对象的创建和使用的问题

  • 6、jQuery社区已经成长为一个生态系统，拥有丰富的插件库

  • 7、学习门槛低，想深入学习也有路径
  • 8、链式操作方式方便，例如要加入队列动画时，只需要像平常一样用;隔开就行，但在JS中—链式动画要很麻烦，甚至有时要用到回调函数
    缺点：
  • 不能向后兼容
  • 插件兼容性

zepto轻量级的jquery

专为移动端的动画库

bootstrap

优点：

  • 最大的优点是确定了DOM结构，
  • 响应式框架设计，解决各种兼容性问题，使得我们对布局、样式的设定变得非常简单，
  • bootstrap4中将选项变量移动到一个个单独的Sass变量中，例如要修改页面的颜色风格，只需要改变样式表中的颜色变量，然后再用sass编译，高度自动化
  • 更重要的是这个框架的思想，例如珊格系统，既符合交互设计中的美学，也使得各个屏幕中的自适应变得有理可循。
  • 将常见的CSS布局小组件和JavaScript插件进行了完整并完善的封装

缺点：
样式千篇一律，能兼容大多数页面需求，但不能满足所有的设计需求，对页面设计有很大局限性

如何提升用户体验

雅虎Best Practices for Speeding Up Your Web Site：

  • content方面

    1. 减少HTTP请求：合并文件、CSS精灵、inline Image
    2. 减少DNS查询：DNS查询完成之前浏览器不能从这个主机下载任何任何文件。方法：DNS缓存、将资源分布到恰当数量的主机名，平衡并行下载和DNS查询
    3. 避免重定向：多余的中间访问
    4. 使Ajax可缓存
    5. 非必须组件延迟加载
    6. 未来所需组件预加载
    7. 减少DOM元素数量
    8. 将资源放到不同的域下：浏览器同时从一个域下载资源的数目有限，增加域可以提高并行下载量
    9. 减少iframe数量

  • Server方面

    1. 使用CDN
    2. 添加Expires或者Cache-Control响应头
    3. 对组件使用Gzip压缩
    4. 配置ETag
    5. Flush Buffer Early
    6. Ajax使用GET进行请求
    7. 避免空src的img标签
  • Cookie方面
    1. 减小cookie大小
    2. 引入资源的域名不要包含cookie
  • css方面
    1. 将样式表放到页面顶部
    2. 不使用CSS表达式
    3. 使用不使用@import
    4. 不使用IE的Filter
  • Javascript方面
    1. 将脚本放到页面底部
    2. 将javascript和css从外部引入
    3. 压缩javascript和css
    4. 删除不需要的脚本
    5. 减少DOM访问
    6. 合理设计事件监听器
  • 图片方面
    1. 优化图片：根据实际颜色需要选择色深、压缩
    2. 优化css精灵
    3. 不要在HTML中拉伸图片
    4. 保证favicon.ico小并且可缓存
  • 移动方面
    1. 保证组件小于25k
    2. Pack Components into a Multipart Document

事件代理

先说事件流
IE的事件流，是采用了事件冒泡机制，
而DOM标准事件流则分为三个阶段:事件捕获阶段，处理事件阶段，事件冒泡阶段。

一个ul里很多li，每个li都有自己编号，当我们要监听所点击的li编号时，如果一个一个li添加事件处理器的话，少的话还好，多的话成百上千问题就出来了，事件处理器将有可能导致内存泄露或者是性能下降———-用得越多这种风险就越大。

所以需要在节点父层监听事件，也就是ul，等待子节点li点击事件冒泡上来，然后获取event.target(所点击li节点)

事件捕获【从父层到子层】，事件冒泡【从子层到父层】

阻止冒泡：ev.stopPropagation();

http头消息

1.通用头标：即可用于请求，也可用于响应，是作为一个整体而不是特定资源与事务相关联。
2.请求头标：允许客户端传递关于自身的信息和希望的响应形式。
3.响应头标：服务器和于传递自身信息的响应。
4.实体头标：定义被传送资源的信息。即可用于请求，也可用于响应。

原生AJAX

ajax！！！

readyState变化时触发onreadystatechange事件，获取xmlhttp.responseText

readyState属性 状态 有5个可取值： 0=未初始化 ，1=启动 2=发送，3=接收，4=完成
0：请求未初始化（还没有调用 open()）。
1：请求已经建立，但是还没有发送（还没有调用 send()）。
2：请求已发送，正在处理中（通常现在可以从响应中获取内容头）。
3：请求在处理中；通常响应中已有部分数据可用了，但是服务器还没有完成响应的生成。
4：响应已完成；您可以获取并使用服务器的响应了。

100-199 用于指定客户端应相应的某些动作。
200-299 用于表示请求成功。
300-399 用于已经移动的文件并且常被包含在定位头信息中指定新的地址信息。
400-499 用于指出客户端的错误。400 1、语义有误，当前请求无法被服务器理解。401 当前请求需要用户验证 403 服务器已经理解请求，但是拒绝执行它。
500-599 用于支持服务器错误。 503 – 服务不可用
